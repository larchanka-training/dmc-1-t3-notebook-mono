# План реализации: AI-процесс в notebook

> Неканонический русскоязычный companion. Каноническая версия: [05-ai-integration-plan.md](./05-ai-integration-plan.md).

## Цель

Реализовать AI-процесс Version 1 внутри notebook так, чтобы пользователь мог получать проверенный `JavaScript` код из исходного `text` block через основной backend-опосредованный путь AI и вставлять результат в notebook как обычный редактируемый `code` block.

## Артефакт задачи

- Текущий утвержденный артефакт задачи: запрос пользователя разложить направление AI-интеграции на основе `docs/ai-architecture.md`, `docs/plans/mvp-roadmap.md` и `docs/plans/sprints.md` и сохранить его как переиспользуемый markdown-план
- Источник по roadmap: `docs/plans/mvp-roadmap.md`, Stage 7, `AI-Assisted Notebook Workflow`
- Канонический контракт AI: `docs/ai-architecture.md`
- Ограничения системной архитектуры: `docs/system_architecture.md`
- Ограничения frontend-архитектуры: `ui/docs/ui_architecture.md`
- Ограничения backend-архитектуры: `api/docs/api_architecture.md`
- Планы по вышестоящим зависимостям:
  - `docs/plans/01-auth-backend-plan.md`
  - `docs/plans/02-notebook-persistence-plan.md`

## Допущения

- Этот план покрывает только первый исполнимый AI-срез Stage 7, а не все AI-идеи, которые сейчас перечислены в `docs/plans/sprints.md`.
- Основной путь AI в Version 1 остается `frontend -> backend API -> provider`, где `AWS Bedrock` используется как главный путь к модели, без прямого рабочего соединения браузера с провайдером.
- `WebLLM` не обязателен для первого поставляемого среза и должен считаться отложенной или дополнительной областью, если команда отдельно не поднимет его в приоритет.
- Backend-хранение notebook, локальная рабочая копия и ручная синхронизация либо уже внедряются, либо достаточно стабильны, чтобы AI опирался на реальную модель notebook и block, а не только на временное состояние редактора.
- Первый срез должен генерировать только `JavaScript` и не должен расширяться до многих языков, сохраняемой истории запросов или нового типа блока `ai`.
- Сгенерированный AI-код остается недоверенным, никогда не выполняется на backend и становится обычным редактируемым содержимым notebook только после вставки на frontend.

## Архитектурные замечания

- Нужно сохранить зафиксированные AI-решения Version 1 из `docs/ai-architecture.md`:
  - AI работает на уровне отдельного блока
  - исходной поверхностью является существующий `text` block
  - backend проверяет запрос перед обращением к модели
  - обычные детерминированные проверки предпочтительнее дополнительных обращений к LLM там, где этого достаточно
  - если извлеченный код не проходит проверку, backend должен ограниченно попросить модель исправить ответ
- Нельзя вводить сохраняемый notebook `ai` block или расширять схему notebook ради истории запросов или ответов.
- Backend должен оставаться в документированной feature-driven структуре: `api/app/features/ai/{router,schemas,service,repository,models}.py` или в эквивалентных внутренних модулях, если это нужно.
- Frontend должен оставаться в зафиксированных FSD-границах: AI-поведение живет в `ui/src/features/ai/`, а встраивание в редактор идет через существующие editor-срезы.
- Поведение по умолчанию для контекста должно соответствовать `scope: this`; более широкий контекст notebook должен быть ограниченным и детерминированным.
- Первый пользовательский срез должен быть минимальным:
  - пользователь пишет или редактирует исходный `text` block
  - пользователь запускает AI-генерацию для этого блока
  - frontend собирает ограниченный контекст и отправляет запрос на backend
  - backend проверяет запрос, обращается к модели, извлекает код, проверяет синтаксис и при необходимости ограниченно просит исправить ответ
  - frontend вставляет полученный код в следующий пустой `code` block или создает новый `code` block после исходного блока
- Выходы выполнения остаются по умолчанию несохраняемыми и не должны незаметно попадать в сохраняемое содержимое notebook через AI-сценарии.

## Что не входит в это направление

- Прямой рабочий доступ браузера к провайдеру
- Сохраняемая история AI-запросов
- Новый тип notebook-блока `ai`
- Широкий контекст всего notebook по умолчанию
- Автоматическая постоянная передача output в AI-контекст
- Сложная схема доработки кода за пределами задокументированного упрощения `convert code to text for AI revision`
- Аналитика стоимости, панели по токенам и сложный выбор провайдера

## Task 9: `T3/BTF -> BACK: Добавить multilingual prompt-policy screening для code-generation intent`

**Описание:** Расширить существующую backend-проверку prompt-policy так, чтобы канонический AI-endpoint принимал запросы на code-generation и code-revision, сформулированные не только на английском, но и на поддерживаемых неанглоязычных языках, начиная с русского, при этом по-прежнему отклоняя non-code intent и unsafe prompt-injection attempts до вызова провайдера.

**Критерии приемки:**
- [ ] Backend prompt screening принимает поддерживаемые русскоязычные code-intent prompts, например запросы создать функцию, компонент, helper или отрефакторить код, без обязательного перевода prompt на клиенте.
- [ ] Backend prompt screening продолжает отклонять non-code prompts вроде объяснения, суммаризации и general-chat запросов как на английском, так и в поддерживаемых русскоязычных формулировках.

## Контрольные точки

1. **Контрольная точка 1: Первый AI-срез определен однозначно**
   У команды есть одна зафиксированная область Stage 7, один контракт и нет путаницы между `Prompt Cell`, сохраняемым AI-контентом, backend-only-подходом и необязательными fallback-идеями.

2. **Контрольная точка 2: Backend AI-путь существует реально**
   Аутентифицированные frontend-запросы доходят до реального backend-endpoint, который проверяет вход, вызывает границу провайдера, проверяет извлеченный код и возвращает нормализованный результат.

3. **Контрольная точка 3: AI работает внутри обычного notebook-сценария**
   Пользователь может получить `JavaScript` из исходного `text` block и увидеть результат, вставленный в правильное место notebook через реальную модель блоков.

4. **Контрольная точка 4: Функция проверяема и эксплуатационно понятна**
   Для успешных и ошибочных AI-сценариев есть проверки на backend и в UI, а у команды есть минимальные инструкции по запуску и сопровождению основного backend AI-пути.

5. **Контрольная точка 5: Prompt policy стала multilingual без расширения области endpoint**
   Backend по-прежнему применяет code-only и unsafe-prompt правила, но поддерживаемые code-intent prompts больше не считаются неявно только англоязычными.

## Риски и меры снижения

- **Риск:** backend-хранение notebook, локальная рабочая копия или синхронизация еще будут меняться во время старта AI-интеграции.
  **Мера снижения:** жестко привязать первый AI-срез к канонической модели notebook и block и не вводить параллельную AI-специфичную модель содержимого.

- **Риск:** `WebLLM` и другие fallback-идеи раздуют область работ до появления рабочего backend-пути.
  **Мера снижения:** явно считать `WebLLM` отложенной задачей до завершения и проверки backend-first-среза.

- **Риск:** качество ответов модели окажется нестабильным и начнет ломать frontend-поведение.
  **Мера снижения:** держать извлечение кода и проверку синтаксиса на backend и возвращать нормализованные ошибки вместо сырого текста модели.

- **Риск:** правила проверки запроса, ограничения размера и приведение ошибок будут недостаточно определены и создадут дыры в безопасности или надежности.
  **Мера снижения:** определить ограничения контракта заранее и включить их в backend-тесты и просмотр DevOps-конфигурации.

- **Риск:** prompt policy будет вести себя так, будто endpoint поддерживает только английский, и начнет ложно отклонять валидные code-запросы на поддерживаемых языках команды.
  **Мера снижения:** добавить явное multilingual prompt-screening coverage как ограниченный backend follow-up без изменения public API contract и без расширения endpoint до general chat.

- **Риск:** QA слишком рано начнет вкладываться в большой каталог запросов до стабилизации первого среза.
  **Мера снижения:** привязать приемочный набор только к реально реализованному поведению и расширять покрытие позже.
