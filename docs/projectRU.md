# Project

## 1. Краткое описание

Этот проект — веб-платформа в формате `JavaScript notebook`.

Продукт объединяет текст, исполняемый код, результаты выполнения, локальное хранение, синхронизацию через `backend` и AI-генерацию кода внутри одного notebook-документа.

Основной пользовательский опыт строится вокруг вертикального документа из упорядоченных блоков. Пользователь пишет заметки, пишет или генерирует `JavaScript`, запускает код шаг за шагом и видит результат прямо в контексте документа.

## 2. Определение продукта

Продукт — это офлайн-ориентированный, block-based `JavaScript notebook` для исполняемых технических документов.

Он предназначен для:

- ведения технических заметок
- пошагового выполнения `JavaScript`
- исследования API и данных
- визуализации результатов
- использования AI для генерации и уточнения кода прямо внутри notebook workflow

## 3. Цели продукта

Продукт должен позволять пользователю:

- создавать и редактировать notebook-документы
- совмещать текст и код в одном месте
- выполнять код по блокам
- сохранять execution state между блоками
- видеть output рядом с кодом, который его породил
- работать локально без постоянной доступности сервера
- вручную синхронизировать состояние notebook с backend
- использовать AI для генерации и уточнения кода по описанию

## 4. Основные принципы

1. `Notebook first`
Главный объект системы — notebook из упорядоченных блоков.

2. `JavaScript first`
Первая версия сфокусирована на `JavaScript`.

3. `Block-based model`
Система строится вокруг notebook blocks, а не вокруг универсальной widget-платформы.

4. `Vertical reading flow`
Основной интерфейс — вертикальный документ, а не free-form canvas.

5. `Offline first`
Локальная работа должна быть возможна даже без доступного backend.

6. `Manual sync`
Синхронизация — это явная функция продукта.

7. `Execution isolation`
Пользовательский код должен выполняться в изолированной среде.

8. `AI inside the workflow`
AI должен помогать создавать и уточнять блоки внутри notebook, а не существовать только как отдельный чат.

9. `Scope discipline`
Сначала продукт должен хорошо решать задачу notebook, и только потом расширяться в соседние категории.

## 5. Канонические понятия

| Термин | Значение |
|---|---|
| `Notebook` | Документ из упорядоченных блоков |
| `Block` | Единица notebook-контента, например текст или код |
| `Text Block` | Блок с plain text или Markdown |
| `Code Block` | Блок с исполняемым `JavaScript` |
| `Output` | Результат выполнения блока |
| `Execution Session` | Runtime-контекст, который хранит переменные и состояние между запусками блоков |
| `Runtime` | Изолированная среда исполнения |
| `Sync` | Явная синхронизация между локальным и серверным состоянием |
| `AI Request` | Пользовательская инструкция для генерации или уточнения кода |

Предпочтительная терминология:

- использовать `block`, а не `widget`, для основных единиц notebook
- использовать `notebook`, а не `page builder`, для модели документа
- использовать `execution session`, когда речь идет о сохраняемом runtime-state

## 6. Основные сценарии использования

### 6.1 Исследовательская работа с кодом

Пользователь пишет `JavaScript` маленькими шагами, запускает выбранные блоки и переиспользует предыдущие результаты.

### 6.2 Исполняемые технические заметки

Пользователь объединяет объяснение, код и output в одном техническом документе.

### 6.3 Исследование API и данных

Пользователь получает данные, преобразует их и визуализирует результат как текст, таблицы или графики.

### 6.4 Легкие внутренние мини-дашборды

Пользователь собирает notebook, который загружает данные и показывает читаемую operational view через код.

### 6.5 AI-assisted prototyping

Пользователь описывает задачу, получает сгенерированный код, правит его и выполняет внутри notebook.

### 6.6 Обучение и onboarding

Пользователь создает материалы, которые объединяют объяснения, примеры и результаты выполнения.

Первая версия должна дать сильное ядро `JavaScript notebook`.

Пользователь должен иметь возможность:

- войти в систему

- открыть и редактировать notebook

## 10. Функциональные требования

### 10.1 Управление notebook

Система должна позволять пользователям:

- создавать notebook
- открывать notebook
- переименовывать notebook
- удалять notebook
- просматривать список notebook
- находить notebook через базовый поиск или фильтрацию

### 10.2 Редактирование блоков

Редактор должен поддерживать:

- добавление блока выше или ниже другого блока
- удаление блока
- перемещение блока вверх или вниз
- редактирование содержимого блока
- сворачивание и разворачивание блоков там, где это полезно

Минимально поддерживаемые типы блоков:

- `Text / Markdown Block`
- `JavaScript Code Block`

### 10.3 Выполнение кода

Модель выполнения должна поддерживать:

- запуск одного выбранного блока
- запуск всех блоков
- запуск от выбранного блока вниз
- общее состояние между блоками в рамках одной `execution session`
- понятную связь между блоком и его output

### 10.4 Output и визуализация

Продукт должен поддерживать notebook-friendly output.

Приоритетные формы output:

- plain text output
- structured object output
- table output
- chart output

### 10.5 AI-assisted workflow

AI workflow должен поддерживать:

1. пользователь пишет описание
2. пользователь запрашивает генерацию кода
3. система возвращает код для нового или обновленного code block
4. пользователь при необходимости редактирует запрос
5. пользователь перегенерирует результат и затем выполняет его

Сгенерированный код должен оставаться редактируемым.

### 10.6 Локальное хранение и офлайн-работа

Система должна:

- сохранять notebook data в `IndexedDB`
- позволять локальное редактирование при недоступном backend
- позволять локальное выполнение при доступном runtime
- сохранять unsynced work до явной синхронизации

### 10.7 Синхронизация

Синхронизация должна быть:

- явной
- notebook-aware
- устойчивой к временной офлайн-работе

### 10.8 Аутентификация и доступ

Первая версия должна поддерживать:

- пользовательскую аутентификацию
- приватные notebook по умолчанию

### 10.9 Экспорт

Система должна поддерживать экспорт хотя бы в один переносимый формат.

Допустимые направления:

- structured JSON-based notebook format
- Markdown-oriented export
- другой project-specific portable format

## 11. Нефункциональные требования

### 11.1 Безопасность

Система должна:

- выполнять пользовательский код в изоляции
- не допускать привилегированный доступ notebook-кода к чувствительным ресурсам
- трактовать generated code как untrusted
- отделять control paths от пользовательского контента

### 11.2 Надежность

Система должна:

- избегать silent data loss
- сохранять локальные изменения при недоступной синхронизации
- достаточно понятно показывать sync-state
- деградировать предсказуемо при недоступности backend или AI

### 11.3 Производительность

Система должна:

- быстро открывать notebook
- сохранять отзывчивость при редактировании
- давать понятную обратную связь при выполнении
- избегать ненужного полного пересчета документа

### 11.4 Поддерживаемость

Архитектура должна тяготеть к:

- четким границам компонентов
- предсказуемой модели данных
- ограниченной скрытой связности
- изолированным зонам ответственности
- тестируемой логике

### 11.5 Воспроизводимость

Поведение notebook должно быть понятным и воспроизводимым за счет:

- ясного execution order
- понятных переходов runtime-state
- прослеживаемой связи между содержимым блока и output

## 12. Высокоуровневая архитектура

Продукт должен быть организован вокруг следующих основных частей:

| Компонент | Ответственность |
|---|---|
| `Frontend` | Notebook editor, block rendering, интеграция с локальным хранилищем, sync UI, представление output |
| `Backend` | Authentication, notebook storage, sync API, access control, при необходимости часть AI-broker логики |
| `Database` | Постоянное хранение пользователей, notebook, метаданных и synchronization state |
| `LLM Layer` | Поддержка генерации кода, browser-side, backend-side или hybrid |
| `Execution Runtime` | Изолированное выполнение `JavaScript` и хранение session state |
| `Execution Orchestrator` | Управление жизненным циклом выполнения и координацией с UI/backend |

## 13. Рекомендуемая доменная модель

Продукт должен строиться вокруг следующих сущностей:

| Сущность | Назначение |
|---|---|
| `User` | Владелец аккаунта |
| `Notebook` | Верхнеуровневый документ |
| `Block` | Упорядоченная единица notebook |
| `BlockContent` | Полезная нагрузка блока, например текст или код |
| `ExecutionSession` | Граница runtime-state |
| `BlockOutput` | Результат выполнения блока |
| `SyncState` | Метаданные синхронизации между локальным и серверным состоянием |
| `ExportArtifact` | Переносимое представление для экспорта |

Рекомендации по моделированию:

- хранить notebook content как структуру блоков
- по возможности отделять долговременный контент от execution state
- не проектировать ядро как generic widget system

## 14. Правила для людей и AI-агентов

При работе над проектом следует исходить из следующих правил:

1. Каноническая модель продукта — `notebook -> ordered blocks -> execution session -> outputs`.
2. Первая версия — `JavaScript-first`.
3. Основной UI — вертикальный notebook flow.
4. Локальное хранение и офлайн-режим обязательны.
5. Manual sync — реальная часть продукта.
6. Runtime isolation — hard requirement.
7. AI должен усиливать работу с notebook, а не подменять ее detached chat-парадигмой.
8. В основной терминологии предпочтителен `block`, а не `widget`.
9. Более простое решение предпочтительно, если оно усиливает notebook core.
10. Не следует добавлять архитектуру, которая нужна только для out-of-scope функций.

При предложении изменений полезно сразу помечать их как:

- `Version 1`
- `Post-Version 1`
- `Out of Scope`

- надежно запускать код шагами

- видеть meaningful output, включая таблицы и графики
- продолжать работу офлайн
- вручную синхронизировать notebook state с backend
- попросить AI сгенерировать код по описанию

Если эти сценарии реализованы хорошо, у проекта будет сильный фундамент.
