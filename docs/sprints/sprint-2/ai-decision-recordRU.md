# AI Decision Record

> Неканонический русскоязычный companion. Каноническая версия: [ai-decision-record.md](./ai-decision-record.md).

## Status

For Team Discussion

## Назначение

Этот документ собирает основные варианты AI-архитектуры, которые стоит обсудить с командой до того, как считать текущую AI-архитектуру окончательно принятой.

Он нужен, чтобы ответить на три практических вопроса из задачи спринта:

1. Как должна работать генерация?
2. Какой результат система должна отдавать на выходе?
3. Должна ли LLM работать только через backend API, только в браузере или в обоих режимах?

Этот документ не заменяет финальную архитектуру. Это decision record, который показывает варианты и объясняет, почему текущая архитектура склоняется к одному из них.

## Что нужно учитывать при обсуждении

Команде стоит оценивать варианты с учетом следующих ограничений:

- Version 1 делает `JavaScript` notebook
- AI должен помогать работе внутри notebook, а не заменять ее отдельным чатом
- доступ backend к Bedrock должен оставаться приватным
- сгенерированный код считается недоверенным
- frontend и backend делают разные люди, поэтому им нужен стабильный контракт
- локальные и demo-сценарии полезны, но поддержка браузеров и устройств отличается

## Какие вопросы нужно решить

### 1. Как должна работать генерация?

Варианты:

1. Прямая генерация без проверок
2. Генерация только с извлечением кода
3. Генерация с извлечением кода, проверкой синтаксиса и ограниченным числом попыток исправления

### 2. Что должно быть результатом на выходе?

Варианты:

1. Сырой текст ответа модели
2. Нормализованная строка с проверенным кодом
3. Новый сохраняемый `ai` block в notebook
4. Код для вставки в обычный `code` block после исходного `text` block плюс служебная metadata

### 3. Где должна работать модель?

Варианты:

1. Только через backend API
2. Только в браузере через `WebLLM`
3. Смешанная схема: backend-first плюс локальный режим в браузере

## Разбор вариантов

### A. Только backend API

Описание:

- frontend отправляет AI-запросы на backend
- backend обращается к Bedrock
- браузер не общается с провайдером напрямую

Плюсы:

- лучшая граница безопасности для ключей и контроля над провайдером
- единые правила логирования, таймаутов и ошибок
- один канонический путь интеграции для frontend и backend
- проще централизованно держать prompt policy и валидацию

Минусы:

- нет локального fallback, если backend недоступен
- для полноценного end-to-end AI flow нужен готовый backend
- хуже подходит для offline/demo-сценариев

Оценка:

- сильный вариант по умолчанию для production
- лучше всего соответствует требованиям спринта по безопасности и DevOps

### B. Только браузерный `WebLLM`

Описание:

- модель работает в браузере
- frontend сам собирает контекст и запускает генерацию
- backend для генерации не нужен

Плюсы:

- не требует выдавать провайдерские ключи клиенту
- полезен для демо и локальных экспериментов
- уменьшает зависимость от сервера для генерации

Минусы:

- поддержка браузеров и устройств нестабильна
- модель может долго скачиваться и тяжело работать
- сложнее централизованно контролировать поведение, логирование и валидацию
- сложнее обеспечить один единый путь интеграции для всей команды

Оценка:

- полезно как эксперимент или demo-path
- слабый вариант как единственное решение для Version 1

### C. Смешанная схема: backend-first и локальный режим

Описание:

- backend-path остается основным
- `WebLLM` используется как опциональный локальный режим или как retry после сбоя backend

Плюсы:

- сохраняется безопасный основной путь
- остаются demo/local/offline-похожие сценарии
- появляется понятная история с fallback

Минусы:

- становится больше продуктовой и UX-сложности
- нужно тестировать два пути выполнения
- результаты Bedrock и локальной модели могут отличаться

Оценка:

- самый сбалансированный вариант, если команда готова к дополнительной сложности
- при этом он должен оставаться именно `backend-first`, а не двумя равноправными режимами

## Варианты пайплайна генерации

### 1. Прямая генерация без проверок

Схема:

- prompt -> model -> return answer

Плюсы:

- быстрее всего реализовать

Минусы:

- слабый контроль качества
- модель может вернуть markdown, пояснения или невалидный код
- плохо соответствует задаче спринта про проверку синтаксиса

Оценка:

- допустимо только как ранняя заглушка
- не подходит как финальный пайплайн Version 1

### 2. Генерация только с извлечением кода

Схема:

- prompt -> model -> extract code -> return code

Плюсы:

- убирает markdown-обертку и лишние пояснения
- остается относительно простой

Минусы:

- невалидный `JavaScript` все еще может пройти дальше
- не закрывает полное backend-требование по валидации

Оценка:

- лучше, чем сырой ответ
- но все еще недостаточно для задачи спринта

### 3. Генерация с извлечением, проверкой и исправлением

Схема:

- prompt -> model -> extract code -> validate syntax
- если код невалиден -> попросить модель исправить один или два раза с текстом ошибки
- вернуть проверенный код или нормализованную ошибку

Плюсы:

- точно соответствует backend-требованию спринта
- лучший баланс качества и реалистичности реализации
- QA-команде проще проверять такой flow

Минусы:

- выше задержка, чем у наивного пайплайна
- больше backend-логики

Оценка:

- рекомендуемый пайплайн для Version 1

## Варианты результата на выходе

### 1. Сырой ответ модели

Плюсы:

- самый простой transport shape

Минусы:

- frontend сам должен убирать пояснения и markdown
- слабый контракт между frontend и backend

Оценка:

- не рекомендуется как основной продуктовый контракт

### 2. Нормализованная строка с проверенным кодом

Описание:

- backend возвращает строку `code` и metadata вроде provider, model, request id и статуса проверки

Плюсы:

- понятный API-контракт
- удобно вставлять во frontend
- хорошо совпадает с notebook model, где код остается обычным редактируемым контентом

Минусы:

- backend берет на себя больше ответственности

Оценка:

- рекомендуемый вариант для Version 1

### 3. Сохраняемый `ai` block

Плюсы:

- явно видно AI-происхождение результата

Минусы:

- появляется новый тип контента в notebook
- усложняются sync, export, порядок блоков и схема документа

Оценка:

- слишком большой scope для Version 1

### 4. Обычный code block плюс optional metadata

Описание:

- исходный `text` block остается канонической notebook-документацией и спецификацией для AI
- сгенерированный `code` block остается каноническим исполняемым контентом
- optional metadata в будущем может хранить последний prompt или provider

Плюсы:

- сохраняет простую notebook model
- позволяет позже добавить provenance без новых типов блоков

Минусы:

- это future scope, а не базовое решение для V1

Оценка:

- хорошее направление для будущего расширения

## Рекомендуемый набор решений

Текущую архитектуру проще всего защитить, если команда согласует такой набор решений:

1. Доступ к модели идет по схеме `backend-first`.
2. `WebLLM` остается опциональным локальным режимом и retry-path, а не основным production-путем.
3. Backend возвращает нормализованный проверенный код, а не сырой текст ответа модели.
4. AI использует существующий `text` block как источник запроса и вставляет сгенерированный код в обычный `code` block после этого `text` block.
5. Backend проверяет prompt, извлекает код, валидирует синтаксис и ограниченно просит модель исправить ответ.
6. Version 1 не требует отдельного пользовательского AI-блока; любое временное AI-состояние остается только техническим UI-state.
7. Optional AI provenance metadata остается future scope, а не частью baseline.

## Почему выбран именно этот вариант

Это направление предпочтительно, потому что оно:

- лучше всего соответствует требованию спринта про Bedrock и приватность backend
- сохраняет простую схему notebook
- использует существующую документацию в notebook как спецификацию для генерации
- дает frontend чистый и понятный контракт
- держит чувствительную логику безопасности на backend
- позволяет позже добавить локальный режим, не делая его основным решением
- закрывает самую сложную часть спринта: извлечение, проверку и исправление невалидного кода
