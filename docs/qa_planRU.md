# Стратегия тестирования API и UI

Документ опирается на [project.md](./project.md) (модель продукта, функциональные и нефункциональные требования Version 1), [system_architecture.md](./system_architecture.md) (потоки, границы компонентов, фиксированные решения V1), [tech_stack.md](./tech_stack.md) (стек и локальные домены) и на инфраструктуру монорепозитория (`docker-compose.yaml`, reverse-proxy в `proxy/` для `notebook.com` / `api.notebook.com` / `pgadmin.notebook.com`).

Репозитории `api/` и `ui/` подключены как git-submodules (`git submodule update --init --recursive` при пустых каталогах). Детальная реализация эндпоинтов и экранов живёт в соответствующих репозиториях; ниже — стратегия уровня продукта и интеграции. Каркас E2E-пакета `e2e/` и контракт прогонов для CI описаны в [tasks.md](../tasks.md).

## 1. Цели и границы

**Цель:** убедиться, что пользователь может пройти сценарии успеха из раздела 15 [project.md](./project.md) и что контракты backend ↔ frontend, безопасность исполнения кода и офлайн-режим не ломаются при регрессиях.

**В фокусе Version 1:**

- аутентификация (`Email + OTP`, `Google OAuth`) и приватность блокнотов по умолчанию;
- CRUD блокнотов, список, поиск/фильтрация (минимум);
- редактор блоков (`text` с Markdown, `code` с JavaScript), порядок, удаление;
- исполнение кода: один блок, все блоки, от выбранного вниз;
- общая `execution session` и привязка вывода к блоку;
- выводы: `text`, `object`, `table`, `chart`, `error` (артефакты рантайма, по умолчанию не входят в durable-состояние блокнота);
- IndexedDB, работа без бэкенда, явный sync без автоматического merge при конфликте;
- экспорт в portable notebook JSON;
- AI: запрос → код → правка запроса → регенерация → подтверждение/редактирование → исполнение.

**Вне фокуса тест-дизайна V1:** совместное редактирование в реальном времени, Python/мультиязычные ядра, LSP, плагины и прочее из раздела 9 [project.md](./project.md).

## 2. Принципы и пирамида тестов

| Уровень | Назначение | Где размещать (ориентир) |
|--------|------------|---------------------------|
| Модульные / юнит | Чистая логика: порядок блоков, merge sync-состояния, сериализация экспорта, валидация DTO | `api` (сервисы, use-cases), `ui` (stores, reducers, утилиты без DOM) |
| Интеграционные API | HTTP + БД + миграции Alembic, политики доступа по пользователю | `api` (test client FastAPI, транзакционная/тестовая БД) |
| Контракт / consumer | Согласованность схем запросов/ответов с фронтом | OpenAPI из FastAPI + тесты клиента или Pact-подобные проверки |
| Интеграционные UI | Компоненты с реальным роутером/state, без полного браузера | `ui` (`Vitest` + Testing Library — см. [tech_stack.md](./tech_stack.md)) |
| E2E (сквозные) | Критические пользовательские потоки под HTTPS и реальными доменами | Пакет `e2e/` (`Playwright`; теги `@smoke`, `@blocks`, `@exec`, `@offline`, `@export`, `@ai`) |

Правило: дорогие E2E покрывают **узкий набор** happy-path и несколько деградаций; ширину сценариев закрывают API и юнит-тесты.

### Виды тестов для issue

В шаблоне бага (`.github/ISSUE_TEMPLATE/bug_report.yml`) чекбоксы «Виды тестов» задают **минимальный набор прогонов до закрытия тикета**. Отмечай только то, что реально нужно по области бага.

| Пункт в issue | Когда отмечать |
|----------------|----------------|
| **Юнит-тесты** | Менялась чистая логика (утилиты, редьюсеры, валидация, merge sync и т.п.) — нужен тест на фикс или регрессию. |
| **Интеграционные API** | Затронуты HTTP, БД, миграции, ответы 401/403, контракт эндпоинта. |
| **Интеграционные UI** | Затронуты компоненты, роутинг, клиентский стейт без необходимости полного браузера. |
| **E2E** | Баг проявляется только в полном потоке пользователя (домены, cookies, сеть, несколько экранов). |
| **Регрессия** | Высокий риск побочных поломок в смежных сценариях V1 — явный прогон области после фикса. |

Если для бага достаточно ручной проверки по шагам из issue без автотестов — не отмечай юнит/API/UI/E2E, оставь только **Регрессию**, если она нужна, или не отмечай ничего и зафиксируй объём в Acceptance Criteria.

## 3. Окружения и данные

| Окружение | Назначение |
|-----------|------------|
| Локально `docker-compose` | API `:8000`, UI `:3000`→`5173`, Postgres `:5432`, pgAdmin `:5050`, proxy `:8080` (HTTP) / `:8443` (HTTPS) на `notebook.com`, `api.notebook.com`, `pgadmin.notebook.com` (см. `proxy/nginx.conf`, [Local-Proxy.md](./Local-Proxy.md)) |
| Прямой доступ без proxy | `http://localhost:3000` (UI), `http://localhost:8000` (API) — для отладки; E2E и auth-сценарии предпочтительно через HTTPS и домены |
| CI | Те же сервисы; `E2E_BASE_URL` (дефолт `https://notebook.com:8443` или согласованный URL стенда); `ignoreHTTPSErrors` для самоподписанного сертификата; для cookie/SameSite — тот же схемный контекст, что в локальной разработке через proxy |

**Тестовые данные:** фикстуры пользователей, пустой и наполненный блокнот, блоки `text` (Markdown) и `code` (JS), длинный документ для перфоманса. Для auth в CI — test doubles: фиксированный OTP, отдельное OAuth test application или заглушка callback; production-секреты и реальную почту не использовать.

## 4. Стратегия тестирования API

Стек по репозиторию: **FastAPI**, **PostgreSQL**, **Alembic**. Ниже — области покрытия, выровненные по сущностям [project.md](./project.md) (раздел 13) и [system_architecture.md](./system_architecture.md).

### 4.1 Аутентификация и сессии

- **Email + OTP:** запрос OTP, доставка (мок внешнего email-сервиса), успешная верификация, неверный/просроченный OTP, повторный запрос.
- **Google OAuth:** старт flow, callback, создание/привязка пользователя, отказ/ошибка провайдера.
- После успешного входа — установка backend-managed secure **`HTTP-only` session cookie** (не bearer-токен в `localStorage`); доступ к ресурсам без cookie → 401; чужой блокнот → 403 (приватность по умолчанию).
- Истечение сессии: граничные значения `SESSION_TTL_SECONDS` (и при наличии отдельного слоя — `TOKEN_TTL_SECONDS`) из env.
- Прокси: корректные заголовки `X-Forwarded-*`, работа за HTTPS на `:8443` (влияние на `Secure` cookie).

### 4.2 Notebook (CRUD и список)

- Создание, чтение, переименование, удаление; список только своих блокнотов.
- Поиск/фильтрация (минимальный scope FR 10.1): регистронезависимость, пустой результат, спецсимволы.

### 4.3 Блоки и контент

- Создание/обновление блоков с типами `text` (содержимое Markdown) и `code` (JavaScript); порядок (`order` или эквивалент).
- Удаление блока; сдвиг порядка остальных.
- Валидация: пустой контент, слишком большой payload (лимиты API), недопустимый тип блока.

### 4.4 Синхронизация (SyncState)

- Явный sync: успешная запись на сервер; идемпотентность повторного sync с тем же телом.
- **Конфликт:** при расхождении локальной и серверной копии backend возвращает conflict **без автоматического merge**; UI показывает явное состояние и ожидает решение пользователя (не сценарий «тихо взять более новую версию»).
- Восстановление после обрыва сети: повтор запроса sync; локальные несинхронизированные правки сохраняются в IndexedDB до успешного sync.
- Соответствие метаданных sync состоянию БД после успешной операции.

### 4.5 AI-брокер (backend-mediated)

- Успешная генерация/рефайн кода для выбранного блока; таймаут/5xx внешнего LLM → предсказуемая ошибка для UI.
- Ответ трактуется как **недоверенный**: на сервере не исполняется; после вставки во фронт — обычный редактируемый контент блока.

### 4.6 Нефункциональные аспекты API

- **Безопасность:** отсутствие SQL-инъекций (параметризованные запросы), ограничение rate (если появится), отсутствие утечки внутренних ошибок в ответах; учётные данные LLM и OAuth только на backend.
- **Надёжность:** транзакции при массовых обновлениях блоков; откат при ошибке.
- **Производительность:** время ответа списка и открытия крупного блокнота (бюджеты зафиксировать в команде).

**Инструменты:** `httpx.AsyncClient` / `TestClient`, фабрики через pytest, БД — отдельная схема или транзакции с откатом; при наличии — snapshot OpenAPI для регрессии схем.

## 5. Стратегия тестирования UI

Стек по репозиторию: **React**, **TypeScript**, **Vite**. Критичны **вертикальный поток блоков**, **IndexedDB**, **ручной sync**, **client-side execution orchestrator** и **изоляция рантайма** (Web Worker / iframe / sandbox — уточнить в `ui`).

### 5.1 Модульные и компонентные тесты

- Рендер типов блоков `text` и `code`; collapse/expand.
- Редактор: ввод, сохранение черновика в локальное состояние, переключение блоков без потери несохранённого (если задумано).
- Индикаторы sync: «нет сети», «есть несинхронизированные изменения», успех sync, **явный conflict** (без silent merge).
- Auth UI: форма email/OTP, редирект Google OAuth (моки сети через MSW).

### 5.2 E2E (Playwright)

Запуск против `https://notebook.com:8443` (после `hosts`, `docker compose` и доверия к самоподписанному сертификату) либо CI-аналог с `E2E_BASE_URL` и `ignoreHTTPSErrors: true`.

**Обязательные сценарии (traceability → FR 8, 10, 15):**

1. Вход (**Email + OTP** или **Google OAuth** — минимум один способ в smoke, оба в полном наборе) → список блокнотов → создание → открытие.
2. Добавить блок `text` (Markdown) и `code`; переместить вверх/вниз; удалить.
3. Выполнить один кодовый блок; выполнить «все»; выполнить «от текущего вниз»; проверить, что переменные из предыдущего блока видны в следующем (`execution session`).
4. Выводы: `text`, `object` (JSON-подобное отображение), `table`, `chart`, **`error`** — структурная/визуальная проверка DOM; после перезагрузки страницы выводы не обязаны восстанавливаться из durable-состояния (повторный run при необходимости).
5. Офлайн: отключить сеть (`page.context().setOffline(true)` / `route.abort`) → редактирование сохраняется в IndexedDB → восстановление сети → **ручной** sync → данные на сервере (проверка через API или повторный вход).
6. **Конфликт sync:** две копии с расхождением → явное состояние конфликта, без автоматического слияния.
7. Экспорт: скачанный **portable notebook JSON** с ожидаемой структурой (метаданные + блоки; без обязательных снимков execution output).
8. AI: описание → генерация → правка промпта → регенерация → **подтверждение/редактирование вставки** → код исполняется.

**Регрессии UX:** фокус-ловушки в редакторе, скролл к выводу после run, длительное выполнение (спиннер/отмена, если есть).

### 5.3 Специфика исполнения пользовательского кода

- Негативные сценарии: бесконечный цикл, `throw`, тяжёлый цикл — тип вывода `error`, сообщение пользователю, не зависание всей вкладки (таймаут worker, где применимо).
- **Изоляция:** попытка доступа к `parent`, `localStorage` приложения, `fetch` к внутреннему API с session cookie — ожидаемое ограничение по модели безопасности (зафиксировать ожидаемое поведение в тест-кейсе после просмотра реализации рантайма).

## 7. Нефункциональное тестирование (сквозно)

- **Производительность:** время до интерактивности списка и редактора; время run-all на N блоках (порог согласовать).
- **Надёжность:** закрытие вкладки с несохранённым локальным состоянием — восстановление notebook content после reopen; execution outputs — только после повторного run, если не кэшируются локально отдельно.
- **Доступность (a11y):** базовые роли для кнопок Run/Sync, подписи у индикаторов ошибок и conflict.
- **Безопасность:** см. 4.6 и 5.3; при появлении CSP — проверка отсутствия регрессий для worker/iframe.

## 8. Критерии готовности к релизу (QA exit)

- Все сценарии раздела 5.2 проходят на эталонном окружении (`docker-compose` + proxy + hosts).
- Нет открытых блокеров по sync, конфликтам и потере данных при обрыве сети.
- API: smoke + критический набор интеграционных тестов зелёный в CI.
- Оба способа входа (OTP и Google) проверены хотя бы в smoke-объёме.
- Документированы известные ограничения (самоподписанный SSL, порт `:8443` для HTTPS локально).

## 10. Следующие шаги для команды

1. Убедиться, что submodules `api` и `ui` инициализированы; зафиксировать актуальный OpenAPI и маршруты.
2. Закрыть инфраструктурные задачи из [tasks.md](../tasks.md) (pytest, Vitest, Playwright `e2e/`).
3. Вынести минимальный **smoke-набор** в CI: API health + `e2e:smoke` (канарейка + login → создать блокнот, когда UI готов).
4. Согласовать бюджеты времени ответа и размера документа для перфоманс-проверок.
5. Привести [Local-Proxy.md](./Local-Proxy.md) в соответствие с `proxy/nginx.conf` (домены `notebook.com`, порты `8080`/`8443`).
