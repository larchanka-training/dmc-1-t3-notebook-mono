# Стратегия тестирования API и UI

Документ опирается на [project.md](./project.md) (модель продукта, функциональные и нефункциональные требования Version 1) и на инфраструктуру монорепозитория (`docker-compose.yaml`, reverse-proxy для `notebook.com` / `api.notebook.com`). Репозитории `api/` и `ui/` подключены как git-submodules: детальная реализация эндпоинтов и экранов живёт в соответствующих репозиториях; ниже — стратегия уровня продукта и интеграции, применимая после инициализации сабмодулей.

---

## 1. Цели и границы

**Цель:** убедиться, что пользователь может пройти сценарии успеха из раздела 15 [project.md](./project.md) и что контракты backend ↔ frontend, безопасность исполнения кода и офлайн-режим не ломаются при регрессиях.

**В фокусе Version 1:**

- аутентификация и приватность блокнотов по умолчанию;
- CRUD блокнотов, список, поиск/фильтрация (минимум);
- редактор блоков (текст/Markdown, JavaScript), порядок, удаление;
- исполнение кода: один блок, все блоки, от выбранного вниз;
- общая `execution session` и привязка вывода к блоку;
- выводы: текст, структурированные данные, таблица, график;
- IndexedDB, работа без бэкенда, явный sync;
- экспорт;
- AI: запрос → код → правка запроса → регенерация → редактируемый код.

**Вне фокуса тест-дизайна V1:** совместное редактирование в реальном времени, Python/мультиязычные ядра, LSP, плагины и прочее из раздела 9 [project.md](./project.md).

## 2. Принципы и пирамида тестов

| Уровень | Назначение | Где размещать (ориентир) |
|--------|------------|---------------------------|
| Модульные / юнит | Чистая логика: порядок блоков, merge sync-состояния, сериализация экспорта, валидация DTO | `api` (сервисы, use-cases), `ui` (stores, reducers, утилиты без DOM) |
| Интеграционные API | HTTP + БД + миграции Alembic, политики доступа по пользователю | `api` (test client FastAPI, транзакционная/тестовая БД) |
| Контракт / consumer | Согласованность схем запросов/ответов с фронтом | OpenAPI из FastAPI + тесты клиента или Pact-подобные проверки |
| Интеграционные UI | Компоненты с реальным роутером/state, без полного браузера | `ui` (Vitest + Testing Library и т.п.) |
| E2E (сквозные) | Критические пользовательские потоки под HTTPS и реальными доменами | Отдельный пакет или `ui/e2e` (Playwright рекомендуется: мультиконтекст, сеть, trace) |

Правило: дорогие E2E покрывают **узкий набор** happy-path и несколько деградаций; ширину сценариев закрывают API и юнит-тесты.

### Виды тестов для issue

В шаблоне бага (`.github/ISSUE_TEMPLATE/bug_report.yml`) чекбоксы «Виды тестов» задают **минимальный набор прогонов до закрытия тикета**. Отмечай только то, что реально нужно по области бага.

| Пункт в issue | Когда отмечать |
|----------------|----------------|
| **Юнит-тесты** | Менялась чистая логика (утилиты, редьюсеры, валидация, merge sync и т.п.) — нужен тест на фикс или регрессию. |
| **Интеграционные API** | Затронуты HTTP, БД, миграции, ответы 401/403, контракт эндпоинта. |
| **Интеграционные UI** | Затронуты компоненты, роутинг, клиентский стейт без необходимости полного браузера. |
| **E2E** | Баг проявляется только в полном потоке пользователя (домены, cookies, сеть, несколько экранов). |
| **Регрессия** | Высокий риск побочных поломок в смежных сценариях V1 — явный прогон области после фикса. |

Если для бага достаточно ручной проверки по шагам из issue без автотестов — не отмечай юнит/API/UI/E2E, оставь только **Регрессию**, если она нужна, или не отмечай ничего и зафиксируй объём в Acceptance Criteria.

## 4. Стратегия тестирования API

Стек по репозиторию: **FastAPI**, **PostgreSQL**, **Alembic**. Ниже — области покрытия, выровненные по сущностям [project.md](./project.md) (раздел 13).

### 4.1 Аутентификация и сессии

- Успешный вход и выдача токена/сессии; истечение `TOKEN_TTL` / `SESSION_TTL` (граничные значения из env).
- Доступ к ресурсам без учётных данных → 401; чужой блокнот → 403 (приватность по умолчанию).
- Прокси: корректные заголовки `X-Forwarded-*`, работа за HTTPS (влияние на secure-cookie, если применимо).

### 4.2 Notebook (CRUD и список)

- Создание, чтение, переименование, удаление; список только своих блокнотов.
- Поиск/фильтрация (минимальный scope FR 10.1): регистронезависимость, пустой результат, спецсимволы.

### 4.3 Блоки и контент

- Создание/обновление блоков с типами text/markdown и code (JavaScript); порядок (`order` или эквивалент).
- Удаление блока; сдвиг порядка остальных.
- Валидация: пустой контент, слишком большой payload (лимиты API), недопустимый тип блока.

### 4.4 Синхронизация (SyncState)

- Явный sync: конфликт «сервер новее / клиент новее», идемпотентность повторного sync.
- Восстановление после обрыва сети (частично — на уровне API повтор запроса с тем же телом).
- Соответствие метаданных sync состоянию БД после успешной операции.

### 4.5 AI-брокер (если на backend)

- Успешная генерация; таймаут/5xx внешнего LLM → предсказуемая ошибка UI.
- Ответ трактуется как **недоверенный**: не выполняется на сервере, только сохраняется/отдаётся как текст для блока.

### 4.6 Нефункциональные аспекты API

- **Безопасность:** отсутствие SQL-инъекций (параметризованные запросы), ограничение rate (если появится), отсутствие утечки внутренних ошибок в ответах.
- **Надёжность:** транзакции при массовых обновлениях блоков; откат при ошибке.
- **Производительность:** время ответа списка и открытия крупного блокнота (бюджеты зафиксировать в команде).

**Инструменты:** `httpx.AsyncClient` / `TestClient`, фабрики через pytest, БД — отдельная схема или транзакции с откатом; при наличии — snapshot OpenAPI для регрессии схем.

## 5. Стратегия тестирования UI

Стек по репозиторию: **Vite**-dev сервер во frontend-контейнере. Критичны **вертикальный поток блоков**, **IndexedDB**, **ручной sync** и **изоляция рантайма** (часто Web Worker / iframe / sandbox — уточнить в `ui`).

### 5.1 Модульные и компонентные тесты

- Рендер типов блоков; collapse/expand.
- Редактор: ввод, сохранение черновика в локальное состояние, переключение блоков без потери несохранённого (если задумано).
- Индикаторы sync: «нет сети», «есть несинхронизированные изменения», успех/ошибка sync.

### 5.2 E2E (Playwright или аналог)

Запуск против `https://notebook.com` (после настройки `hosts` и доверия к сертификату) либо CI-аналог с теми же заголовками.

**Обязательные сценарии (traceability → FR 8, 10, 15):**

1. Вход → список блокнотов → создание → открытие.
2. Добавить текстовый блок (Markdown), кодовый блок; переместить вверх/вниз; удалить.
3. Выполнить один кодовый блок; выполнить «все»; выполнить «от текущего вниз»; проверить, что переменные из предыдущего блока видны в следующем (execution session).
4. Выводы: plain text, объект (JSON-подобное отображение), таблица, график — визуальная/структурная проверка DOM.
5. Офлайн: отключить сеть (Playwright route.abort / offline) → редактирование сохраняется в IndexedDB → восстановление сети → ручной sync → данные на сервере (проверка через API или повторный вход).
6. Экспорт: файл скачан / буфер с ожидаемой структурой.
7. AI: описание → генерация → правка промпта → регенерация → код остаётся редактируемым и исполняется.

**Регрессии UX:** фокус-ловушки в редакторе, скролл к выводу после run, длительное выполнение (спиннер/отмена, если есть).

### 5.3 Специфика исполнения пользовательского кода

- Негативные сценарии: бесконечный цикл, `throw`, тяжёлый цикл — корректное сообщение об ошибке, не зависание всей вкладки (где применимо — таймаут worker).
- **Изоляция:** попытка доступа к `parent`, `localStorage` приложения, `fetch` к внутреннему API с украденными куками — ожидаемое ограничение по модели безопасности (зафиксировать ожидаемое поведение в тест-кейсе после просмотра реализации рантайма).

## 7. Нефункциональное тестирование (сквозно)

- **Производительность:** время до интерактивности списка и редактора; время run-all на N блоках (порог согласовать).
- **Надёжность:** закрытие вкладки с несохранённым локальным состоянием — восстановление после reopen (если заявлено).
- **Доступность (a11y):** базовые роли для кнопок Run/Sync, подписи у индикаторов ошибок.
- **Безопасность:** см. 4.6 и 5.3; при появлении CSP — проверка отсутствия регрессий для worker/iframe.

## 8. Критерии готовности к релизу (QA exit)

- Все сценарии раздела 5.2 проходят на эталонном окружении.
- Нет открытых блокеров по sync и потере данных при обрыве сети.
- API: smoke + критический набор интеграционных тестов зелёный в CI.
- Документированы известные ограничения (например, самоподписанный SSL локально).
